# The content-drafting layer is an optional v4 add-on, behind an output-plugin interface

research-tool's drafting layer — `draft.py` (multi-format publish bundles: LinkedIn / blog / Substack / threads / newsletter / notes), `voice_lint.py` + `voice_tighten.py`, the `essays`/`cityfront` content angles, and `claim_lint.py` — is **Santiago-specific content generation, not generic research tooling**. It stays **out of Inkwell core**: core ends at synthesis + recommendation (v3). Drafting ships as an **optional v4 add-on**, gated behind an **output-plugin interface** so a generic Inkwell user never compiles or sees it, while Santiago (and anyone who wants content output) can enable it.

This mirrors research-tool's own `GENERALIZE.md` Phase B ("plugin output stage"): the pipeline ends at a recommendation, and each user selects their output(s) — briefing doc, evidence ledger, FOIA drafter, or, here, social/blog drafts. Inkwell adopts the same seam: `internal/output/` defines an interface; the drafting implementation is one plugin behind a build tag or config flag. Prompts in `research-tool/prompts/formats/` and the voice config carry over to the plugin verbatim.

v4 is **a real track, not an indefinite defer** — but it is explicitly *last* and *optional*: it must not pull core sprints, and core (v1–v3) must be fully usable without it. The voice/angle logic is the heaviest port (it encodes one person's taste); it is the part most likely to stay thin or stay in Python even after the interface exists.

**Considered and rejected**: putting drafting in core (forces every user to carry content-generation machinery irrelevant to investigators/researchers — the generic audience Inkwell targets); deleting it / leaving it in Python forever (throws away working, valuable output and the natural test of whether the synthesis layer produces draft-worthy material); building the output-plugin interface now in v1–v3 (premature — there is exactly one consumer until v4; design the seam when the second output type is real, per the same logic as ADR-0011's stance on provider abstraction).
