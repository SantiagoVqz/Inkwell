# SQLite access: sqlc-generated queries over the pure-Go modernc driver

Inkwell talks to SQLite through **sqlc**-generated type-safe Go, backed by the **pure-Go `modernc.org/sqlite`** driver. SQL lives in hand-written `.sql` files (schema + named queries); sqlc reads them at build time and emits plain `database/sql` code that is checked into the repo. sqlc is a build-time code generator with zero runtime footprint — no ORM, no query builder, no reflection. Malformed SQL fails code generation, not production.

The driver choice is dictated by ADR-0009 (distribution is first-class): `modernc.org/sqlite` is a pure-Go transpilation of SQLite, so the binary stays statically linkable and cross-compiles to every target OS without a C toolchain. The cgo-based `mattn/go-sqlite3` is faster (it is the real C engine) but would drag a per-target C cross-compilation toolchain into the goreleaser pipeline — reintroducing exactly the install friction the Go rewrite exists to remove. At v1 ingest volumes (hundreds of rows per run) the performance gap is irrelevant; revisit only if profiling ever shows SQLite as a bottleneck.

**Considered and rejected**: `mattn/go-sqlite3` (cgo breaks clean cross-compilation and the single-static-binary promise of ADR-0009; speed advantage is moot at v1 volumes); a runtime ORM such as GORM or ent (hides SQL, fights the "SQL stays SQL" learning goal of ADR-0001, and adds a runtime dependency for no benefit at this scale); hand-written `database/sql` with no codegen (loses compile-time type safety on query results and column mapping — the main thing sqlc buys, for the cost of a tool we already committed to in the roadmap).
